How I Wish Matt's Presentation Would Be

June 2, 2014

This post reflects my point of view on Wordpress as a product and its development. It's a subjective review of the presentation and does not represent MalesBanget.com ( or it might )

Last night, I attended the 3rd Wordpress Jakarta Meetup at Binus University, Jakarta. Starring the famous Matt Mullenweg himself, I put a lot of hopes for opening my perspective about Wordpress future. Unfortunately, I was sadly mistaken.<!-- more -->

The Opening

Matt at 3rd Jakarta Wordpress Meetup

Matt opened the event with the history of how Wordpress evolved. This is my first time, watching his appearance and I must say that he's very realistic yet visionary kind of person. But, I'm not here for that. I'm here for more, to hear the plan how Wordpress will conquer the web, for at least more than 22% sites in the whole world.

There were few things that Matt only introduced a little about JetPack and mobile enhancement. Most of them were answered during the Q&A sessions. There's none on the 1/4 width of screen, displaying the presentation, but it was hardly move. I encouraged to raise my hand, but not selected.  So, here's what I would like to hear, Matt.

What I Would Like To Hear

In this era of REST API and web openness, Wordpress needs to separate the product into 3 parts:

  1. Backend Administration ( CMS )

  2. Frontend

  3. REST API

I know the Codex has been there for a very long time, but as a developer, I don't want to be entangled with Wordpress guidelines to build a website. I'm pretty sure that the companies would more consider to hire a all around PHP developer instead of Wordpress developer. Therefore, a REST API would be pretty good.

Then comes JetPack, bundled with many kind of features, including the JSON API. At first, I thought it is a solution for me to build a website using JetPack's API. But, it lacks of features and doesn't come with optional framework for developer to build a custom API.

Second thing is JSON API should be resided on the website's own platform. But, as far as I could read ( plus used it for 3 days ), we should connect to your domain https://public-api.wordpress.com/ instead of our own domain. I do worry about the load and access time during the connection to your domain, as local connection is much more preferred.  Even though that you are using SSL, but as a developer ( and I think you will understand it ), our own ego forces us to get the data from our own domain. Not to mention that it would a cross domain call for AJAX based web application.

Enough for JetPack.

Caching System

Then I want to talk about the web performance. For few past years, we rely on SuperCache, a fascinating caching system that's ironically not from Wordpress. Why?

Static is The New Dynamic

I really want to support Wordpress and would be very happy to see your development plan, but honestly, I think this is a very fundamental issue that you would need to cover. There are plenty of CMS that we could more rely on, Ghost and AnchorCMS are two of examples.

I know that this event was organised by WPJkt , a great thanks for them. But, not all of the attendees are students, some of them are developers and we are keen to know your plan so we could still use Wordpress as a strategic product partner and not only because you have quite large portion of market share.

So, Matt. Would you do a rerun for your presentation in Jakarta?

Update: This event was hosted by WPJkt, not Binus University. Thanks for Valent for the implicit hint.
